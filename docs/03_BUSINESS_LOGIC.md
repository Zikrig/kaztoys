**Проект:** Dara Exchange

## 1. Состояния пользователя (FSM)

Бот ведёт пользователя по сценариям. Ниже — группы состояний и переходы.

### 1.1 Общие состояния

| Состояние | Описание |
|-----------|----------|
| `MAIN_MENU` | Пользователь в главном меню; получает рассылку новых заявок и уведомления. |
| (нет состояния) | После таймаута 10 мин или кнопки «В главное меню» — сброс в `MAIN_MENU`. |

### 1.2 Онбординг (только для новых пользователей)

- После `/start`: показать дисклеймер → видео → «Готовы начать поиск?» → кнопка «Начать поиск».
- Переход в флоу поиска (опрос или показ заявки).

### 1.3 Создание заявки (ListingCreate)

| Шаг | Состояние FSM | Что ждём | Переход дальше |
|-----|----------------|----------|----------------|
| 1 | `ListingCreate.wait_photo` | Фото с кодом | → wait_category |
| 2 | `ListingCreate.wait_category` | Выбор категории (кнопка) | → wait_age |
| 3 | `ListingCreate.wait_age` | Выбор возраста (кнопка) | → wait_description |
| 4 | `ListingCreate.wait_description` | Текст (описание + на что менять) | → confirm |
| 5 | `ListingCreate.confirm` | «Подтвердить» / «Изменить» | Готово или → wait_photo (тот же код) |

Проверка подписки: перед созданием заявки (или перед показом формы) — если нет активной подписки, предложить купить доступ на 2 недели; при отказе → главное меню.

### 1.4 Редактирование заявки (ListingEdit)

Тот же цепочка, что и создание, но код уже есть (не генерируем новый). Состояние можно общее с созданием, в данных FSM храним `listing_id` или `code`.

### 1.5 Поиск (Search)

| Шаг | Состояние FSM | Что ждём | Переход |
|-----|----------------|----------|---------|
| Опрос (если фильтры не сохранены) | `Search.wait_category` | Категория или «Все категории» | → wait_age |
| | `Search.wait_age` | Возраст | → wait_district |
| | `Search.wait_district` | Район или «Любой район» | Сохраняем фильтры, → showing |
| Показ заявки | `Search.showing` | Кнопка: Предложить обмен / Дальше / В главное меню; или «Изменить параметры поиска» | В зависимости от нажатия |
| После «Предложить обмен» | (флоу отклика) | После отправки отклика: «Продолжить поиск?» | Да → следующая заявка (Search.showing); Нет → MAIN_MENU |
| Заявки кончились | `Search.no_results` | «Изменить параметры» / «В главное меню» | Опрос или MAIN_MENU |
| Подсказка после 7 «Дальше» | `Search.showing` + счётчик | «Изменить параметры» / «Продолжить поиск» | Счётчик сброса подсказки за сессию |

**Счётчики в контексте поиска (в FSM или в памяти):**
- `search_skip_count` — сколько раз подряд нажато «Дальше» без отклика; при 7 — показать подсказку один раз, затем сбросить или не показывать снова за сессию.
- `search_offset` — смещение по списку заявок для «следующая заявка».

### 1.6 Отклик (Response)

| Шаг | Состояние FSM | Что ждём | Переход |
|-----|----------------|----------|---------|
| Выбор игрушки | `Response.choose_listing` | «Выбрать имеющуюся» (показать список заявок пользователя) / «Создать новую» | Список с кнопками «Отправить эту» или → create_new |
| Создание нового отклика | `Response.wait_photo` | Фото с кодом | → wait_description |
| | `Response.wait_description` | Текст (описание + сообщение продавцу) | → confirm |
| | `Response.confirm` | «Подтвердить отправку» | Отправка отклика, уведомление автору заявки → выход в меню или «Продолжить поиск?» |

Контекст: в FSM храним `target_listing_id` (на какую заявку откликаемся), при «Предложить обмен» из поиска — также сохраняем, чтобы после отправки спросить «Продолжить поиск?».

### 1.7 Просмотр «Мои заявки» / отклики по заявке

- Не обязаны держать отдельное состояние: пользователь нажал «Посмотреть отклики» по заявке X — показываем отклики, под каждым callback «Выбрать» с `response_id`.
- При нажатии «Выбрать» — создаём мэтч, отправляем уведомления обеим сторонам. Состояние можно сбросить в MAIN_MENU или оставить в «просмотре заявок» с кнопкой «Вернуться в меню».

### 1.8 Мэтч и подтверждение сделки

- Отдельного FSM не требуется: приходят сообщения с кнопками «Подтвердить сделку» / «Сделка не состоялась» (callback с `match_id` и `action`).
- Обработчик: обновление записи в `matches` (подтверждение или отмена). Если оба подтвердили — начисление +1 к `confirmed_deals` обоим пользователям.

## 2. Правила и инварианты

### 2.1 Заявки

- У пользователя может быть несколько открытых заявок (по ТЗ — «до 20 заявок» как вариант; лимит можно задать константой).
- Код заявки уникален глобально (среди заявок и откликов).
- При закрытии заявки: статус `closed`; если кто-то уже нажал «выбрать» по этой заявке — ему отправить «Пользователь удалил заявку».

### 2.2 Отклики

- Один отклик привязан к одной заявке и одному пользователю.
- У одного отклика может быть не более одного мэтча (один раз «выбран»).
- Код отклика уникален глобально.

### 2.3 Мэтчи

- Мэтч создаётся один раз при нажатии «Выбрать» по отклику. Повторное нажатие по тому же отклику не создаёт второй мэтч.
- Подтверждение: оба нажали «Подтвердить сделку» → статус `both_confirmed`, обоим +1 к счётчику подтверждённых сделок.
- «Сделка не состоялась» — хотя бы один нажал; можно считать мэтч отменённым, счётчик не увеличиваем.

### 2.4 Поиск

- Заявки в выдаче: только `status = open`.
- Сортировка: по `created_at` DESC (свежие первыми).
- Фильтры: категория (или «все»), возраст, район (или «любой»). Соответствие: точное по полям заявки; для «любой возраст»/«любой район» в фильтре — не накладываем условие на это поле.
- Показ по одной: запрос с LIMIT 1 OFFSET N; N увеличивается при «Дальше», сбрасывается при смене фильтров или новом входе в поиск (по желанию можно не сбрасывать до «Изменить параметры»).

### 2.5 Подписка

- Активная подписка: `expires_at > NOW()`.
- Без активной подписки: не может создавать заявки (предлагаем купить); при необходимости ограничить и просмотр поиска (по ТЗ — уточнить). В главном меню рассылка новых заявок — «если включена подписка» (по плану).

### 2.6 Таймаут неактивности

- 10 минут без действий (любого сообщения/нажатия от пользователя) → сброс состояния в `MAIN_MENU`, отмена текущего сценария (создание заявки, опрос поиска и т.д.).

## 3. Сервисы и их обязанности

| Сервис | Ответственность |
|--------|-----------------|
| **user** | Регистрация/обновление по telegram_id; получение/обновление профиля; сохранение и чтение search_filters; увеличение confirmed_deals. |
| **listing** | Генерация уникального кода; создание/обновление/закрытие заявки; список открытых заявок пользователя; получение заявки по id. |
| **response** | Создание отклика (код, фото, описание); привязка к заявке; список откликов по заявке; пометка «выбран» при создании мэтча. |
| **search** | Получение следующей заявки по фильтрам пользователя (category, age_group, district) с пагинацией; сохранение фильтров; (этап 2) подписка на уведомления и отправка при новой подходящей заявке. |
| **match** | Создание мэтча при «Выбрать»; запись подтверждений; проверка «оба подтвердили» и начисление confirmed_deals. |
| **subscription** | Проверка активной подписки; создание подписки на 2 недели; возврат срока действия. |

## 4. События и уведомления

| Событие | Кто получает | Действие |
|---------|--------------|----------|
| Новая заявка создана | Все пользователи в главном меню с активной подпиской | Рассылка карточки заявки с кнопкой «Отправить предложение». |
| На заявку поступил отклик | Владелец заявки | Сообщение «НА ВАШУ ЗАЯВКУ ПОСТУПИЛ ОТКЛИК» + 2 фото + кнопка «Откликнуться»/«Принять». |
| Ваш отклик выбрали | Автор отклика | «ВАШ ОТКЛИК ВЫБРАЛИ» + контакт владельца заявки + кнопка «Связаться в телеге». |
| Создан мэтч | Оба участника | «ПОМНИТЕ???» (2 фото + текст заявки и отклика) + кнопки «Подтвердить сделку» / «Сделка не состоялась». |
| Оба подтвердили сделку | Оба | Сообщение об успехе; в профиле +1 подтверждённая сделка. |
| Заявка закрыта | Пользователь, который уже нажал «выбрать» по этой заявке | «Пользователь удалил заявку». |
| (Этап 2) Новая подходящая заявка | Пользователи, долиставшие до конца поиск с сохранёнными фильтрами | «Новая игрушка в категории …, возраст …, район …» + кнопка «Посмотреть». |

## 5. Справочники (константы)

- **Категории (v4):** Самокаты, велики и транспорт; Для малышей (0–2 года); Куклы, фигурки и наборы; Машинки и стрелялки; Другое (домики, горки и т.д.). В коде: slug или ключ для БД.
- **Возраст:** 0–2 года, 3–5 лет, 6–8 лет, 9–12 лет, Любой возраст.
- **Районы:** список Астаны — хранить в коде (keyboard) или в таблице `districts`; при изменении списка — правка в одном месте.

Эти справочники используются в создании заявки (категория, возраст), в поиске (фильтры) и в отображении карточки заявки (категория, возраст, район).
